## Overview

This lab separates traffic entry, application runtime, and data persistence into different Linux hosts. The goal is not only to make services run, but to practice production-like responsibility boundaries, access control, service lifecycle management, failure isolation, and operational validation.

The system is composed of:

- A reverse-proxy host running nginx as the only HTTP entry point.
- An app-server running a Flask application through Gunicorn.
- A `lab-app.service` unit supervised by systemd.
- A db-server running PostgreSQL 16 for persistent application data.
- Health checks, database checks, and backup/restore validation to prove the deployment works beyond process startup.

The application was originally containerized with Docker Compose. This version intentionally uses a VM-style Linux service model to practice the operational concerns that appear when processes, users, configuration, networking, and database permissions are managed directly on hosts.

## Architecture Core

The lab is organized around three operational boundaries.

### Traffic Boundary

The reverse proxy is the only HTTP entry point. Clients do not talk directly to the application server. This centralizes routing, request headers, access logs, and future TLS termination while keeping backend runtime details behind nginx.

### Runtime Boundary

The application server owns the application runtime. The Flask app is served by Gunicorn and supervised by systemd. Runtime configuration is externalized through `/etc/lab-app.env`, and the process runs as the restricted Linux service account `appuser`.

### Data Boundary

The database server owns persistent data. PostgreSQL access is restricted by role, database, source address, and authentication method. Recovery is validated through backup and restore testing instead of assuming that a dump file is sufficient.

## Three-Layer Design

### Reverse Proxy Layer

nginx runs on the reverse-proxy host and accepts client traffic on port 80. Its upstream target is the application server:

```nginx
location / {
    proxy_pass http://app-server:8000;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
}
```

The backend app server is not exposed directly to clients. This layer can later support TLS termination, centralized access logs, rate limiting, request routing, and simple load balancing without changing the application process.

### Application Layer

The app-server owns the runtime. `lab-app.service` is managed by systemd, Gunicorn serves the Flask WSGI application, and Python dependencies are isolated in a virtual environment under the application directory.

```ini
[Service]
User=appuser
Group=appuser
WorkingDirectory=/opt/lab-app
EnvironmentFile=/etc/lab-app.env
ExecStart=/opt/lab-app/venv/bin/gunicorn --bind 0.0.0.0:8000 app:app
Restart=on-failure
RestartSec=3
NoNewPrivileges=true
PrivateTmp=true
```

Runtime configuration is not hardcoded into the application:

```bash
FLASK_ENV=production
DB_HOST=db-server
DB_PORT=5432
DB_NAME=labdb
DB_USER=labuser
DB_PASSWORD=replace-with-secret
```

`appuser` is the Linux service account on app-server. `labuser` is the PostgreSQL role on db-server. They are intentionally separate identities in different security domains.

### Database Layer

PostgreSQL stores persistent application data. The app connects to `labdb` as `labuser`, not as a superuser. Access is constrained in `pg_hba.conf` by source IP, database, user, and authentication method:

```text
# TYPE  DATABASE  USER     ADDRESS              METHOD
host    labdb     labuser  192.168.100.21/32    scram-sha-256
```

Table ownership and grants matter because a successful login does not guarantee object-level access:

```sql
ALTER TABLE notes OWNER TO labuser;
ALTER SEQUENCE notes_id_seq OWNER TO labuser;
GRANT ALL PRIVILEGES ON TABLE notes TO labuser;
```

Backup and restore validation proves the data path works end to end:

```bash
pg_dump -h db-server -U labuser -d labdb -Fc -f labdb.dump
createdb -h db-server -U labuser labdb_restore
pg_restore -h db-server -U labuser -d labdb_restore labdb.dump
psql -h db-server -U labuser -d labdb_restore -c "select count(*) from notes;"
```

## Key Architecture Decisions

| Decision | Reason |
| --- | --- |
| Use a reverse proxy | Centralizes the HTTP entry point, avoids direct client access to the app server, and prepares the design for TLS termination, logging, routing, and load balancing. |
| Use systemd and Gunicorn | systemd practices Linux service lifecycle management, while Gunicorn serves the WSGI application as a long-running runtime process. |
| Externalize environment config | `/etc/lab-app.env` separates runtime configuration from application code and avoids hardcoding database connection settings. |
| Separate `appuser` and `labuser` | `appuser` is a Linux service account on app-server. `labuser` is a PostgreSQL role on db-server. They belong to different security domains. |
| Control access with `pg_hba.conf` | PostgreSQL access is controlled by more than a password: source IP, database name, username, and authentication method all matter. |
| Separate `/health` and `/ready` | `/health` proves the app process is alive. `/ready` proves the app can serve traffic, including database availability. |
| Validate backup and restore | A backup is not useful until restore is tested and application-level data can be queried after recovery. |

## Validation

| Command | What it proves |
| --- | --- |
| `curl http://localhost:8000/health` | The application process is alive on the app-server. |
| `curl http://localhost:8000/ready` | The app can serve readiness checks and reach its database dependency. |
| `curl http://localhost:8000/db` | PostgreSQL connectivity works from the application runtime. |
| `curl http://localhost:8000/notes` | The app can read real rows from the `notes` table. |
| `curl http://app-server:8000/` from reverse-proxy | The reverse-proxy host can reach the app-server upstream. |
| `curl http://localhost/` from reverse-proxy | nginx routes local requests to the upstream application. |
| `curl http://192.168.100.11/` | The reverse-proxy IP entry path works from a client network path. |

```bash
sudo nginx -t
sudo systemctl reload nginx
```

## Failure Domains and Troubleshooting

| Failure Domain | Symptom | Root Cause | Diagnostic Command | Fix |
| --- | --- | --- | --- | --- |
| Service discovery / systemd | Unit `myapp.service` or `lab-app.service` not found | systemd unit file missing or wrong service name | `systemctl status lab-app` | Create `/etc/systemd/system/lab-app.service` and run `systemctl daemon-reload`. |
| Runtime lifecycle | Service exits successfully but port 8000 is not listening | `python app.py` only defined Flask routes and did not start a server | `journalctl -u lab-app`; `ss -tulpn \| grep 8000` | Use Gunicorn in `ExecStart`: `gunicorn -b 0.0.0.0:8000 app:app`. |
| Runtime environment | Service fails due to missing environment variables or file | `EnvironmentFile` not found or incorrect path | `journalctl -u lab-app` | Create `/etc/lab-app.env` and restart the service. |
| Linux service account | Permission denied for `/home/appuser` | `appuser` was created as a restricted service account without a normal home directory | `journalctl -u lab-app` | Create `/var/lib/lab-app` and set `Environment=HOME=/var/lib/lab-app`. |
| Database access control | `no pg_hba.conf entry` | PostgreSQL did not allow the app-server source IP, database, and user combination | `curl /db`; PostgreSQL error message | Add a `pg_hba.conf` rule for `labdb` / `labuser` from the app-server IP. |
| Database identity | Password authentication failed or role missing | `labuser` did not exist or the password mismatched | `\du` in `psql` | Create `labuser` and align `/etc/lab-app.env`. |
| Database ownership / grants | Permission denied for table `notes` | Table was owned by `postgres`, not `labuser` | `\dt`; `SELECT * FROM notes;` as `labuser` | Run `ALTER TABLE notes OWNER TO labuser;` and `ALTER SEQUENCE notes_id_seq OWNER TO labuser;`. |
| Reverse proxy path | External request fails but app-server works locally | nginx upstream path or hostname resolution issue | `nginx -T`; `getent hosts app-server`; `curl http://app-server:8000/` | Correct `proxy_pass` or host resolution. |

## What I Learned

- Three-tier architecture is about responsibility boundaries, not just three VMs.
- A reverse proxy centralizes traffic entry and hides backend runtime details.
- systemd makes the application runtime observable and recoverable.
- PostgreSQL security is layered: role, database, source IP, authentication method, and object permissions all affect access.
- Health checks should distinguish process liveness from service readiness.
- Backup must be validated through restore and an application-level query.

## Next Improvements

- Add a systemd timer for scheduled database backups.
- Add log rotation or structured logging.
- Add TLS termination at the reverse proxy.
- Add monitoring and alerting for service state, readiness, and database availability.
- Add a CI/CD deployment flow.
- Automate restore validation.
- Add an optional Docker vs systemd comparison page.

## 我學到的事

- Three-tier architecture 的重點是 responsibility boundaries，不只是三台 VM。
- reverse proxy 集中 traffic entry，並隱藏 backend runtime details。
- systemd 讓 application runtime 具備 observable 與 recoverable 的操作模型。
- PostgreSQL security 是 layered：role、database、source IP、authentication method 與 object permissions 都會影響 access。
- health checks 應該區分 liveness 與 readiness。
- backup 必須透過 restore 與 application-level query 驗證。
